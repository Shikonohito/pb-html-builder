## 1. Текущее состояние репозитория

На момент анализа в рабочем дереве нет готовой .NET solution, проектов src/*.csproj
или тестовых проектов. Репозиторий содержит:

- docs/ - vision-документы с продуктовой и архитектурной спецификацией;
- example_theory/ - HTML/CSS/JS прототип student-facing theory.html;
- example_practice/ - HTML/CSS/JS прототип student-facing practice.html;
- LICENSE.txt и .gitignore.

Следовательно, этот документ описывает целевую архитектуру, которую нужно создать и
поддерживать. Прототипы example_theory и example_practice следует рассматривать как
UX/reference samples для generated HTML и student runtime, но не как финальный
источник истины для модели данных или artifact generation.

## 2. Назначение продукта

PbHtmlBuilder - локальный single-user builder для Windows. Он запускается как
localhost browser app и позволяет Author создавать учебные HTML-файлы для Student.
Основная первая поставка - полный author+student flow для theory.html.

Роли:

- Author создает и редактирует учебные HTML, answer-key JSON и verified reports.
- Student открывает generated HTML, читает материал, запускает код и отвечает на
  вопросы или assessment tasks.

Отдельной роли Reviewer нет. Verified review считается author-режимом.
Authentication и authorization не требуются. Граница доступа - локальные файлы,
working directory, answer-key JSON и процесс передачи файлов.

## 3. Платформа и app model

Целевая платформа:

- .NET / ASP.NET Core / Blazor target framework: net10.0.
- Builder UI: Blazor Web App с Interactive Server rendering.
- Standalone WebAssembly для builder-а не используется.
- Product shell WebView2 не входит в выбранную архитектуру.
- Host запускается как localhost browser application.
- Приложение Windows-oriented, но authoring flow не должен полагаться на native
  Windows file dialogs.

ASP.NET Core host должен использовать modern hosting model:

- WebApplication.CreateBuilder(args);
- DI registration через builder.Services;
- appsettings/options для конфигурации;
- endpoint mapping;
- короткий и читаемый Program.cs.

## 4. Главные архитектурные принципы

1. Generated HTML является главным источником истины для повторного открытия
   проекта в builder-е. DOM не является source of truth: builder импортирует
   manifest, валидирует schema и реконструирует typed model.

2. Document model строго typed и schemaVersioned. Все редактируемые сущности имеют
   stable IDs. Display numbers вычисляются из порядка и не используются как identity.

3. Rich text хранится как Typed AST, а не как raw HTML. Renderer отвечает за
   escaping, encoding и выпуск безопасного static DOM.

4. Artifact generation выполняется typed C# renderers. Razor templates могут
   использоваться для Blazor UI, но не являются выбранным механизмом генерации
   theory/practice/assessment/verified HTML.

5. Domain остается чистым. Он не ссылается на Blazor, JS interop, file system,
   HTTP, ASP.NET Core host types или HTML string rendering.

6. Document state и authoring commands живут в C#. JS используется как thin
   adapter для browser/runtime APIs: Monaco, Pyodide worker, IndexedDB,
   BroadcastChannel, selection/range measurement, drag/drop details,
   beforeunload prompt и File System Access API для assessment runtime.

7. Backend является authority для Save/Open. Все save/open targets разрешаются
   относительно app working dir. Absolute paths, parent traversal, Windows
   reserved names, invalid chars и resolved path outside root запрещены.

8. V1 реализует Theory полностью. Practice, Assessment и Verified Assessment
   должны быть архитектурно зарезервированы и видимы на стартовой странице как
   disabled/coming soon до реализации.

## 6. Project dependency rules

Разрешенные ссылки:

```text
PbHtmlBuilder.Host
  -> PbHtmlBuilder.UI
  -> PbHtmlBuilder.Application
  -> PbHtmlBuilder.Infrastructure
  -> PbHtmlBuilder.Artifacts

PbHtmlBuilder.UI
  -> PbHtmlBuilder.Application
  -> PbHtmlBuilder.Domain

PbHtmlBuilder.Application
  -> PbHtmlBuilder.Domain

PbHtmlBuilder.Artifacts
  -> PbHtmlBuilder.Domain
  -> PbHtmlBuilder.Application contracts

PbHtmlBuilder.Infrastructure
  -> PbHtmlBuilder.Application
  -> PbHtmlBuilder.Domain

PbHtmlBuilder.Domain
  -> no project references
```

Запрещенные связи:

- Domain -> Blazor/UI/JS interop/file system/HTTP/renderers;
- UI -> concrete file system implementation;
- Artifacts -> UI components;
- generated student runtime -> builder backend calls;
- generated student HTML -> AI/Grammar/authoring-only services;
- Assessment HTML -> answer-key contents.

## 7. PbHtmlBuilder.Host

Host содержит только composition root, конфигурацию, DI и endpoints.

Основные файлы и папки:

```text
src/PbHtmlBuilder.Host/
  Program.cs
  appsettings.json
  appsettings.Development.json
  Configuration/
    BuilderOptions.cs
    WorkingDirectoryOptions.cs
    RuntimeAssetOptions.cs
    DependencyVersionOptions.cs
    BrandDefaultsOptions.cs
  Endpoints/
    FolderEndpoints.cs
    ImportEndpoints.cs
    HealthEndpoints.cs
  ServiceCollectionExtensions.cs
```

Program.cs должен оставаться коротким:

- AddRazorComponents().AddInteractiveServerComponents();
- register Application services;
- register Artifacts renderers;
- register Infrastructure adapters;
- bind options from appsettings;
- map static assets;
- map Razor components;
- map minimal endpoints for folder picker/import where browser-facing API needed.

Конфигурация должна покрывать:

- working dir root behavior;
- local asset base URL для development;
- GitHub Pages runtime asset base URL для export;
- pinned Monaco version/source;
- pinned Pyodide version/source;
- preferred port;
- timeout defaults;
- brand/copyright defaults.

Dev launch:

- official launch через scripts/Start-LocalServer.ps1;
- stop через scripts/Stop-LocalServer.ps1;
- script печатает final localhost URL;
- script не открывает browser автоматически;
- port policy: preferred port из config, fallback на следующий доступный.

## 8. PbHtmlBuilder.Domain

Domain - чистое ядро документа. Оно хранит типы, IDs, schema, validation contracts
и command contracts, но не знает о UI, file system и HTML rendering.

### 8.1. Identity

Папка:

```text
src/PbHtmlBuilder.Domain/Ids/
```

Рекомендуемые value objects:

- DocumentId;
- SectionId;
- BlockId;
- QuestionId;
- AnswerOptionId;
- CodeBlockId;
- TestId;
- MapCellId;
- SubmissionId, если понадобится отдельная identity для submission records.

IDs должны быть короткими random URL-safe identifiers с type prefix. Примеры:

- doc_x7f3k9;
- sec_q2p8ab;
- blk_mn74rc;
- opt_w2z19k;
- test_j8hd3q.

Новый элемент всегда получает новый ID. Удаленный ID может вернуться только через
undo. Display number "01", "02" вычисляется из порядка sections/tasks.


Целевые типы:

- PracticeDocument.
- AssessmentDocument.
- VerifiedAssessmentDocument.
- DocumentMetadata.
- TargetFileMetadata.
- BrandSettings.
- CopyrightSettings.

Root fields:

- documentId;

Сериализация должна быть стабильной. Для polymorphic blocks/documents нужно заранее
выбрать явный discriminator, например "kind" или "blockType", чтобы migrations и

### 8.3. Rich text AST

Папка:

```text
src/PbHtmlBuilder.Domain/RichText/
```

Типы:

- RichTextDocument;
- RichTextInline;
- TextRun;
- InlineCodeRun;
- LinkRun или LinkMark;
- RichTextMark: Bold, Italic, Underline, InlineCode;
- RichTextRange;
- RichTextSelection;
- RichTextNormalizer.

Rich text нужен для paragraphs, list items, table cells, answer options, feedback
и verified comments. Heading text для Monaco title, special h3 и question h3
поддерживает plain text и inline code.

Domain не хранит contenteditable HTML. Paste из внешних источников в V1 вставляется
как plain text by default.

### 8.4. Content blocks

Папка:

```text
src/PbHtmlBuilder.Domain/Blocks/
```

Базовые типы:

- ContentBlock;
- BlockContainer;
- ParagraphBlock;
- ListBlock;
- ListItem;
- ListKind: Ordered, Unordered;
- TableBlock;
- TableRow;
- TableCell;
- MediaBlock;
- MediaKind: Image, VideoOrIframe;
- SpecialBlock;
- SpecialBlockVariant;
- MonacoCodeBlock;
- CodeBlockSettings;
- QuestionBlock.

Rules:

- Section body содержит blocks.
- Content nesting - one-level.
- Section может содержать special block и question block.
- Body special/question block может содержать paragraph/list/table/media/Monaco.
- Nested special/question blocks внутри special/question body запрещены.
- Flat-only model не выбран, recursive arbitrary blocks не выбран.

SpecialBlockVariant для Theory:

- Definition;
- Important;
- Hint;
- CorrectApproach;
- IncorrectApproach.

Practice/Assessment используют variants:

- Important;
- Hint;
- CorrectApproach;
- IncorrectApproach.

MediaBlock хранит external URL. Builder не скачивает и не встраивает media в HTML.
Author считается trusted, поэтому arbitrary links и iframe URLs разрешены.

### 8.5. Sections and map

Папки:

```text
src/PbHtmlBuilder.Domain/Documents/
src/PbHtmlBuilder.Domain/Content/
```

Типы:

- Section;
- TheorySection;
- PracticeTaskSection;
- AssessmentSection;
- SectionMap;
- SectionMapCell.

Theory section map:

- fixed 2x3 grid из шести cells;
- левая часть labels: "Разделы темы" и "Что важно рассмотреть";
- правая часть - editable cell titles и anchors;
- anchor выбирается из текущих section headings;
- если section удалена, anchor сбрасывается или помечается invalid;
- количество map cells не меняется Author-ом в V1.

### 8.6. Questions

Папка:

```text
src/PbHtmlBuilder.Domain/Questions/
```

Типы:

- QuestionKind: SingleChoice, MultipleChoice, TextInput;
- QuestionBlock;
- ChoiceQuestion;
- ChoiceOption;
- ChoiceFeedback;
- TextInputQuestion;
- AcceptedTextAnswer;
- QuestionFeedback.

Theory behavior:

- Single choice default options count: 4.
- More than one correct option превращает label в multiple answer.
- Choice options имеют rich text и optional feedback.
- Text input uses raw exact comparison against configured correct answers.
- Attempts retryable, no persistence between refresh.

Assessment behavior:

- correctness feedback в assessment.html отсутствует;
- multiple-choice toggle updates submission immediately;
- text input accepted through "Ответить";
- correct answers хранятся только в answer-key JSON.

Monaco block fields:

- title;
- code;
- runEnabled;
- readonly;
- debugEnabled для Practice/Assessment;

Practice/Assessment autotests:

- input/print: stdin и expected output, raw exact;

### 8.8. Assessment and verified models

Папки:

```text

src/PbHtmlBuilder.Domain/Verified/
```

Assessment types:

- AssessmentDocument;
- GradableItem;
- GradableItemKind;
- CodeAnswerItem;
- AnswerKeyDocument;
- AnswerKeySection;
- SubmissionDocument;
- SubmissionAnswerRecord;
- ChoiceSubmission;
- TextSubmission;
- CodeSubmission.

Rules:

- assessment section содержит exactly one gradable item;
- assessment HTML не содержит correct answers или grading keys;
- answer-key JSON содержит ключи, tests, expected code и visibility flags;
- submission JSON содержит latest accepted answers by stable IDs;
- submission JSON не содержит student identity fields.

Verified types:

- VerifiedAssessmentDocument;
- VerifiedSourceMetadata;
- VerifiedSectionResult;
- GradeStatus: Correct, Incorrect, PendingReview;
- AutoGradeResult;
- ManualOverride;
- ReviewComment;
- VerifiedSummary.

Rules:

- verified-assessment.html создается из assessment HTML + answer-key JSON +
  submission JSON;
- review builder показывает source assessment content readonly;
- Author может редактировать только comment и manual status override;
- manual override wins over automatic grading;
- saved verified HTML self-contained for report purposes;
- verified manifest хранит source IDs, но не local file paths.

Типы:

- ValidationResult;

- stable IDs uniqueness;
- section map anchors;

- assessment exactly one gradable item;

- relative path constraints;
- manifest integrity before open/save/export.

## 13. Generated student runtime

Generated HTML must not call builder backend or telemetry endpoints. It may load:

- own CSS/JS assets from GitHub Pages latest;
- Monaco and Pyodide from pinned configured sources;
- external media URLs;
- iframe URLs entered by Author.

Runtime architecture:

- lazy-load Monaco on code block focus/visibility;
- lazy-load Pyodide on first Run/Debug;
- Pyodide runs in Web Worker;

- no implicit shared namespace between blocks;

## 14. Artifact-specific architecture

### 14.1. Theory V1

Theory is the only required complete V1 builder.

V1 acceptance must cover:

- create theory document;
- edit all supported block types;
- save HTML;
- open HTML back in builder;
- undo/redo;
- recover from autosave;
- open exported theory.html as file://;
- read material;
- run runnable code blocks;
- answer theory questions;
- golden sample lesson using all block types.

Student layout:

- top bar;
- sidebar;
- main content;
- section map;
- lesson sections.

Top bar:

- sidebar show/hide;
- topic heading area;
- topic-kicker default "Python Bootcamp";
- h1 documentTitle;
- light/dark theme button.

Sidebar:

- title "Оглавление";
- hide button;
- section list with display numbers and titles;
- footer copyright default "© Ivan Ivanov, 2026";
- active item updates through scroll observer;
- collapsed state does not persist.

Theory content:

- section map fixed 2x3;
- sections with computed display number "01";
- blocks: paragraph, ordered list, unordered list, table, image URL, video/iframe
  URL, special block, Monaco code block, question block;
- special body and question body can contain simple content and Monaco, but not
  nested special/question;
- simple tables only: optional header, add/remove rows/columns, rich text cells,
  no merged cells, no sort, no formulas.

Theory questions:

- single answer;
- multiple answer;
- text input answer;
- feedback rich text;
- retryable attempts;
- no persistence between refresh.

### 14.2. Practice future

Practice is future generated artifact. It should guide architecture but not block
V1 Theory.

Key requirements:

- single active task navigation;
- topbar task buttons with status dot;
- sidebar status per task: pending/correct/incorrect;
- task body uses same content model except question blocks should not be used;
- Monaco supports reset/run/copy/debug;
- if debug active, run becomes active;
- at most one debug-enabled Monaco block per section;
- autotest editor appears when debug active;
- test types: input/print, function, class/scenario;
- hidden checkbox exists, but hidden tests are stored openly in practice.html;
- no persistence between sessions; refresh resets code/output/status.

Architecture impact:

- Domain needs reusable Autotest model now, even if Practice UI is disabled.
- SharedEditor should keep Monaco/debug flags extensible.
- Runtime should expose practice status updates as session-local state only.

### 14.3. Assessment future

Assessment is future generated artifact for collecting student answers without
exposing answer keys.

File model:

- assessment HTML: student-facing tasks and manifest without keys;
- answer-key JSON: Author sidecar with correct answers, tests and expected code;
- submission JSON: Student answers.

Rules:

- assessment.html must not contain correct answers or grading keys;
- answer-key JSON is plain JSON protected by process control;
- submission JSON has no student identity fields;
- default sidecar suffixes: .assessment.answer-key.json and
  .assessment.submission.json;
- exactly one gradable item per assessment section;
- student file handle is requested on first accepted answer;
- if File System Access API is unavailable, assessment shows blocked state;
- no manual export/import fallback in selected UX.

Architecture impact:

- SaveAssessmentUseCase must write both assessment HTML and answer-key JSON.
- Assessment renderer and answer-key serializer must be separate outputs.
- Submission schema must be validated as untrusted input.
- Verified matching must use documentId, schemaVersion compatibility and stable IDs.

### 14.4. Verified Assessment future

Verified Assessment is a derived artifact generated from assessment HTML,
answer-key JSON and submission JSON.

Creation flow:

- start page asks for output folder, output filename, title, assessment HTML,
  answer-key JSON and submission JSON;
- builder validates existence/import, metadata, documentId matches, schema
  compatibility and section/item ID matching;
- validation failure blocks generation with report;
- after validation, builder runs grading and opens review builder before Save;
- direct generation without review is not selected.

Review builder:

- assessment content readonly;
- original task text, choices, code answer and student answer not editable;
- Author can add/edit rich text comment per section;
- Author can mark section Correct, Incorrect or PendingReview;
- manual override wins over automatic grading;
- reviewer identity/audit info not stored.

Verified HTML:

- follows assessment layout;
- top summary correct/total;
- sidebar status Correct/Incorrect/PendingReview;
- section border and status use text+icon+color;
- hidden tests do not reveal input/expected details;
- expected code shown only when answer-key visibility flag allows it;
- report opens without answer-key JSON.

## 15. File naming and metadata conventions

Default generated filenames:

- lesson.theory.html;
- lesson.practice.html;
- lesson.assessment.html;
- lesson.assessment.answer-key.json;
- lesson.assessment.submission.json;
- lesson.verified-assessment.html.

Save As must create a new documentId. This prevents old answer keys/submissions

Every generated HTML must contain:

## 16. Main data flows

### 16.1. New Theory flow

1. StartPage loads recovery/recent registries from IndexedDB.
2. Author clicks New Theory.
3. Create form collects relative folder path, filename and document title.
4. Application validates target metadata.
5. Document factory creates TheoryDocument with new documentId.
6. DocumentStore opens unsaved document.
7. TheoryBuilderPage renders from DocumentStore state.
8. First file write happens only on Save.

### 16.2. Edit flow

1. UI component captures an intent: text edit, add block, reorder, toggle flag.
2. UI dispatches a typed command to DocumentStore.
3. Store validates focus/scope and applies command.
4. Store updates document state, dirty state, command history and selection state.
5. Store notifies UI.
6. AutosaveCoordinator schedules snapshot as needed.

### 16.3. Save flow

1. SaveProjectUseCase asks DocumentStore for current document.
2. Validation runs against document and target path.
3. If target file exists, overwrite confirmation is required.
4. If risky save/overwrite/migration, BackupFileService creates timestamped backup.
5. Artifact renderer produces static HTML + manifest.
6. IProjectFileStore writes inside working dir only.
7. RecentProjectRegistry updates.
8. Dirty state clears and toast feedback appears.

### 16.4. Open flow

1. Author chooses Open from project context menu.
2. If file is outside working dir, ImportProjectUseCase copies it into root.
3. ArtifactReader extracts manifest from HTML.
4. Manifest validation and migrations run.
5. Invalid metadata blocks edit with report.
6. Valid document loads into DocumentStore.
7. RecentProjectRegistry updates.

### 16.5. Autosave recovery flow

1. AutosaveCoordinator writes full compact manifest snapshots to IndexedDB every
   5 minutes.
2. IndexedDB keeps 10 rotating slots.
3. Start page lists recent autosave entries.
4. Author opens a recovery entry.
5. Snapshot loads into builder with dirty state preserved.
6. Explicit Save writes generated HTML.

### 16.6. Generated runtime flow

1. Student opens generated HTML, including file:// in Edge/Chrome.
2. Static DOM is readable immediately.
3. Runtime bootstrap reads compact manifest/config.
4. Monaco loads only when code block is visible/focused.
5. Pyodide worker starts on first Run/Debug.
6. Code runs in fresh namespace with timeout and input prompt queue.
7. Runtime failure affects only interactive block, not the whole document.

## 20. Architecture risks and guardrails

Risk: separate builders reduce coupling but can duplicate editor mechanics.
Guardrail: keep section/block mechanics in SharedEditor and Domain reusable models.

Risk: Interactive Server keeps state central but localhost connection loss can
interrupt authoring.
Guardrail: autosave, dirty state and recovery are mandatory.

Risk: backend-only working-dir writes are safer but require clear relative path UX.
Guardrail: in-app folder picker, strict validation, import/copy flow for outside files.

Risk: GitHub Pages latest for own runtime can change old generated HTML behavior.
Guardrail: schemaVersion adapters, metadata stamps and backward-compatible runtime.

Risk: Monaco and Pyodide are heavy.
Guardrail: lazy mount/load in builder and generated HTML.

Risk: assessment submission JSON is not trusted.
Guardrail: verified grading validates IDs/schema and treats JSON as user-controlled input.

Risk: future Grammar Check and AI autocomplete can add privacy, latency and cost.
Guardrail: optional authoring services only, disabled by config, no generated HTML
dependency and no credentials in artifacts.

## 22. Maintenance rules

When changing architecture:

- update this document and the relevant vision document together;
- preserve dependency direction;
- add schema migrations for manifest changes;
- add or update golden snapshots for renderer changes;
- update browser E2E when user-facing authoring flow changes;
- keep generated HTML backward compatibility explicit;
- do not put transient authoring UI state into manifest unless accepted by Author;
- do not move file IO policy into UI components;
- do not let JS become the owner of document state.

The core invariant is simple: C# typed model and command pipeline own authoring
state; typed C# renderers produce artifacts; thin TypeScript adapters handle browser
runtime edges; generated HTML remains readable and self-describing through its
static DOM plus compact manifest.
