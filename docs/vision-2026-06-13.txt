Заданные ограничения и цели

1. Инструмент ориентирован на Windows.

2. Builder является локальным одиночным инструментом с одним активным проектом и одной рабочей вкладкой/поверхностью.

3. Builder должен позволять собирать определённые HTML-страницы по принципу WYSIWYG.

4. Основной формат распространения builder: portable app folder для Windows.

5. Технологическая основа builder: C# backend, Blazor frontend, WebView2 shell.

6. Генерируемые HTML-файлы предназначены для студентов, которые изучают Python.

7. HTML-файлы должны оставаться лёгкими и удобными для пересылки по сети. Целевой ориентир для обычного generated HTML без внешних Monaco/Pyodide/runtime-файлов: до 1-2 MB.

8. Есть три основных типа HTML-файлов: theory, practice и assessment. Verified assessment является readonly-представлением результата проверки assessment.

9. Theory и practice не хранят student state между сессиями. Assessment хранит ответы студента между сессиями и должен поддерживать перенос результата на другое устройство через обновлённый HTML-файл.

10. Для отображения и редактирования code-блоков используется Monaco Editor.

11. Для запуска Python-кода используется Pyodide.

12. Стили и runtime-скрипты generated HTML подтягиваются с GitHub Pages. Во время разработки builder может подключать runtime с local static server.

13. Generated HTML открывается студентом обычным double-click в Edge/Chrome через `file://`.

14. Vision фиксирует product + technical spec. Детальные code-level JSON schema, имена C# классов и layout конкретных компонентов не являются частью этого документа.



Decision summary

1. Builder распространяется как portable Windows app folder и запускает Blazor UI внутри WebView2.

2. Builder поддерживает только один активный проект. При попытке открыть другой проект с unsaved changes показывается prompt `Save / Discard / Cancel`.

3. Состояние документа хранится в Central Document Store. Все изменения проходят через explicit command objects.

4. Undo/redo строится на command model. Структурные действия являются отдельными шагами, набор текста группируется по debounce/blur.

5. JS interop используется только для browser APIs и тяжёлых browser-интеграций: Selection API, DnD helpers, File System Access API, Monaco Editor, Pyodide wrappers.

6. Domain logic, document model, validation, save/open, artifact generation и grading rules остаются в C#.

7. Source of truth для повторного открытия проекта: generated HTML с pre-rendered DOM и полным readable project metadata JSON.

8. Metadata в v1 не сжимается, даже если это дублирует часть pre-rendered контента. Приоритет reopen: exact authoring fidelity.

9. Builder открывает только файлы со строгим PB manifest: `appId`, `documentType`, `documentId`, `schemaVersion`, `runtimeVersion`, ids, `generatedAt`, content hash.

10. Поддерживается только текущая версия схемы. Старые schemaVersion не мигрируются автоматически в v1.

11. Runtime HTML является thin HTML: общие CSS/JS, Monaco, Pyodide и feature modules загружаются извне.

12. Runtime ссылки pinned к конкретной версии GitHub Pages assets. Для собственных CSS/JS используется SRI where possible.

13. Runtime делится на core + feature modules. Monaco загружается lazy only if code exists. Pyodide загружается on first run/debug.

14. Theory и practice сохраняются как single HTML. Assessment сохраняется как student HTML + teacher key JSON.

15. Teacher key JSON хранится рядом с assessment HTML по умолчанию как `same-stem.answers.key.json`, но путь можно переопределить.

16. Student assessment HTML не содержит путь, имя файла, правильные ответы, автотесты или expected code из teacher key JSON.

17. Student assessment HTML содержит только project metadata и dedicated answer state block.

18. Stable GUID/ULID используются для document, sections, content blocks и graded items. Id не меняется при переименовании и перестановке.

19. Pairing assessment HTML и teacher key JSON выполняется по `documentId + contentHash`.

20. В одной секции practice допускается ровно одна graded task. Остальные Monaco-блоки в секции являются example/self-run.

21. Practice поддерживает graded code/debug tasks. Question-блоки в practice не входят в v1.

22. Theory question-блоки являются self-check без persistence.

23. Multi-answer question считается верным только при exact set match: выбран полный набор правильных вариантов и нет лишних.

24. Code-задача считается верной только если все автотесты прошли. Timeout, runtime error и failed expected считаются неверным результатом.

25. Missing answer, missing key или mismatch block id в verified assessment даёт статус `Ожидает проверки`.

26. Assessment student feedback показывает только `Ответ принят`. Correctness раскрывается только в verified assessment.

27. Student assessment сохраняет accepted answers в HTML on submit. Drafts до `Ответить` сохраняются только локально в browser storage.

28. Если File System Access API недоступен или пользователь не дал доступ к записи, runtime показывает persistent status banner и предлагает скачать updated HTML.

29. WYSIWYG feel является non-negotiable даже в первом vertical slice.

30. Builder UI использует inline editing + inspector. Сложные настройки блоков доступны в inspector и/или inline раскрытиях.

31. Ключевые операции должны быть доступны не только через hover/right-click, но и через явные кнопки на selected block.

32. Accessibility target: keyboard access для ключевых действий, видимый focus, ARIA labels для icon buttons.

33. Generated student-facing UI на русском. Builder может использовать смешанный EN/RU для технических команд.

34. Theme policy: system default + local override для builder и generated runtime.

35. No telemetry. Локальные ошибки пишутся в root builder logs directory.

36. Raw HTML/JS в контенте запрещён. Контент создаётся только через поддержанные блоки и sanitized rich-text.

37. URL policy: разрешены безопасные протоколы. Для media нужен `https`; для ссылок допустимы `http`, `https`, `mailto`. `javascript:` и похожие опасные схемы запрещены.

38. First milestone: full vertical slice с core blocks + one graded flow, real save/open, real Pyodide, student assessment submit, verified creation и score.



Shared architecture and artifact model

1. Builder shell:

   1. Builder запускается как Windows desktop-приложение на WebView2.

   2. Builder не требует серверной платформы, учётных записей или облачного storage.

   3. Builder хранит настройки профиля автора локально: course/kicker, author, copyright year, last folders, recent projects, theme preference.

   4. Recent projects отображаются на стартовой странице и используются только как локальное удобство автора.

2. Document model:

   1. У каждого документа есть `documentId`, `documentType`, `schemaVersion`, `runtimeVersion`, `createdAt`, `updatedAt`, `generatedAt`.

   2. У каждой секции, content block, Monaco block, question block, graded item и anchor target есть stable GUID/ULID.

   3. Порядковые номера `01`, `02`, `03` вычисляются из текущего порядка секций и не являются identity.

   4. Заголовки, labels, anchors и статусы отображаются из document model, а не из случайного DOM parsing.

   5. Model должна позволять добавлять новые block types без переписывания базовой структуры документа.

3. Generated HTML shape:

   1. Generated HTML содержит pre-rendered content DOM, чтобы страница быстро показывала учебный контент и сохраняла читаемость при частичных runtime-сбоях.

   2. Generated HTML содержит readable project metadata JSON в dedicated script block:

      `script type="application/json" id="pb-project"`.

   3. Assessment HTML дополнительно содержит answer state block:

      `script type="application/json" id="pb-answer-state"`.

   4. Verified assessment HTML дополнительно содержит review state block:

      `script type="application/json" id="pb-review-state"`.

   5. Runtime не должен использовать DOM как единственный source of truth для graded data, ids, settings, tests или answer state.

4. Project metadata:

   1. `pb-project` хранит полный authoring state, достаточный для exact reopen в builder.

   2. `pb-project` хранит document settings, sections, blocks, rich-text AST, Monaco defaults, allowed packages, runtime options, anchors, validation metadata и public settings.

   3. `pb-project` не хранит правильные ответы, автотесты или expected code для assessment.

   4. Для practice `pb-project` может хранить practice autotests, потому что practice tests являются открытыми. `Hidden` скрывает детали только в UI и не является security boundary.

5. Teacher key JSON:

   1. Teacher key JSON создаётся только для assessment.

   2. Teacher key JSON содержит `documentId`, `contentHash`, `schemaVersion`, generated metadata, correct choices, accepted text answers, code autotests, expected code и grading config.

   3. Teacher key JSON не передаётся студенту.

   4. Student HTML не содержит имя файла или путь к teacher key JSON.

   5. При `Save` assessment builder атомарно сохраняет student HTML и teacher key JSON.

6. Answer state:

   1. `pb-answer-state` содержит только student-submitted accepted answers.

   2. Accepted answer для choice question хранит выбранные option ids.

   3. Accepted answer для text input хранит введённый текст.

   4. Accepted answer для code task хранит код студента.

   5. `pb-answer-state` содержит timestamps submit/update where useful, но не хранит правильные ответы.

   6. Drafts до `Ответить` не входят в `pb-answer-state`; они могут временно храниться в localStorage/IndexedDB.

7. Review state:

   1. Verified assessment HTML содержит review state: auto results, manual overrides, comments, final section statuses и score summary.

   2. Manual override wins over auto-result, но original auto-result сохраняется в metadata.

   3. Teacher comment хранится как limited rich-text: inline marks, link, inline code. Полные nested blocks в комментариях v1 не требуются.

8. Artifact naming:

   1. Theory: один `*.html`.

   2. Practice: один `*.html`.

   3. Assessment: `*.html` для студента и `*.answers.key.json` для преподавателя.

   4. Verified assessment: новый readonly `*.html`, содержащий review-state.

   5. New document flow автоматически добавляет `.html`, если автор ввёл имя без расширения.

   6. Builder валидирует forbidden chars, конфликт существующего файла и Windows Unicode paths.

9. Content hash:

   1. `contentHash` вычисляется по оцениваемой структуре assessment: ids graded items, types, correct-answer mapping references, test definitions references, и relevant grading settings.

   2. Текстовые правки, которые не меняют graded structure, могут не менять `contentHash`, если это явно поддержано реализацией.

   3. Если `documentId` совпадает, но `contentHash` не совпадает, verified creation блокируется или требует явного teacher confirmation.



Shared builder UX and editing model

1. Общий принцип:

   1. Builder должен ощущаться как WYSIWYG editor, а не как form-based CMS.

   2. Основные content changes выполняются inline.

   3. Сложные настройки блока доступны в inspector panel и, где уместно, в inline раскрытии под блоком.

   4. Inspector не заменяет WYSIWYG, а дополняет его для редких/сложных настроек.

2. Selection и block actions:

   1. При наведении на секции, content blocks и элементы side navigation показываются границы области взаимодействия.

   2. При выборе блока показывается компактная панель действий: add before/after, move, duplicate where supported, delete, block settings.

   3. Right-click context menu остаётся доступным для Project actions и block actions.

   4. Hover insertion line с круговой plus-кнопкой используется для добавления секций и блоков между существующими элементами.

   5. Drag-and-drop используется для перестановки секций и блоков. При drag показывается ghost/placeholder над или под целевым элементом.

   6. DnD должен иметь keyboard alternative для accessibility baseline.

3. Sections:

   1. Если секций нет, главная часть показывает круговую plus-кнопку для создания первой секции.

   2. Секция имеет title row: computed number в формате `01` и editable h2 title.

   3. Side navigation автоматически повторяет порядок секций.

   4. Изменение section title сразу обновляет соответствующий side navigation item.

   5. Удаление секции доступно через context menu и selected-block action.

   6. При удалении секции builder предупреждает, если внутри есть graded item или несохранённые student/review-related settings.

4. Rich-text editing:

   1. Rich-text хранится как Inline AST: текстовые spans + marks.

   2. Поддерживаемые marks для body text: bold, italic, underline, inline code, link.

   3. Поддерживаемые marks для h3 block title и Monaco title: inline code.

   4. Rich-text editing реализуется через custom Blazor editor + Selection API.

   5. При выделении текста появляется floating toolbar.

   6. Right-click по выделению открывает text context menu с теми же командами.

   7. Горячие клавиши: Ctrl+B, Ctrl+I, Ctrl+U, Ctrl+Q для inline code, Ctrl+K для link.

   8. Если курсор стоит на слове без выделения, hotkey применяет mark к текущему слову, где это технически надёжно.

   9. Link command открывает inline field для URL. URL проходит sanitization.

5. Undo/redo:

   1. Ctrl+Z отменяет последнее command-level действие.

   2. Ctrl+Shift+Z возвращает отменённое действие.

   3. Набор текста группируется по debounce/blur.

   4. Add/delete/move block, change block type, change graded settings, save path changes и answer-key changes являются отдельными undo steps.

   5. Undo/redo stack является частью editor session, но не сохраняется в generated HTML.

6. Project context menu:

   1. Right-click в builder открывает context menu.

   2. В submenu `Project` доступны `Save`, `Save as...`, `Open...`, optional `Check links/media`, и recovery-related actions where useful.

   3. `Save` сохраняет в существующий путь, если путь установлен.

   4. `Save as...` открывает Windows file dialog.

   5. `Open...` открывает HTML project только при валидном PB manifest.

7. File dialogs:

   1. Builder поддерживает Windows Unicode paths, пробелы и кириллицу.

   2. Last used folders запоминаются в builder profile.

   3. Builder не должен требовать ASCII-only имена файлов.

8. Language:

   1. Generated student-facing UI на русском.

   2. Builder может использовать EN/RU mixed labels: `New`, `Create`, `Save`, `Answer`, `Run`, `Debug`, `Readonly`, `Project`.

   3. Технические сущности в документации могут оставаться на английском: `runtime`, `metadata`, `DocumentStore`, `key JSON`, `answer-state`.

9. Accessibility:

   1. Все icon buttons имеют tooltip и aria-label.

   2. Все важные действия доступны с клавиатуры.

   3. Focus state всегда видимый.

   4. DnD-only operation не должна быть единственным способом выполнить обязательное действие.

10. Theme:

   1. Builder и generated HTML используют system default как initial theme.

   2. Пользовательский override сохраняется локально.

   3. Автор не настраивает custom CSS или custom theme в v1.



Shared generated HTML runtime model

1. Loading model:

   1. Generated HTML ссылается на pinned runtime version на GitHub Pages.

   2. Dev generated HTML может ссылаться на local static server runtime assets.

   3. Runtime делится на core bundle и feature modules.

   4. Core runtime отвечает за theme, navigation, metadata reading, answer-state handling, basic interactivity и error surfaces.

   5. Monaco module загружается только если документ содержит code-блоки.

   6. Monaco instances инициализируются lazy при попадании блока в viewport или при первом interaction с блоком.

   7. Pyodide загружается only on first run/debug.

2. File mode:

   1. Student HTML должен работать при открытии через `file://` double-click в Edge/Chrome.

   2. Runtime не должен требовать локального HTTP server для обычного студента.

   3. Если browser policy блокирует часть возможностей, UI должен объяснить ограничение и предложить поддерживаемый fallback.

3. Runtime load failures:

   1. Если core runtime не загрузился, pre-rendered content должен оставаться видимым where possible.

   2. Если Monaco/Pyodide не загрузился в theory, practice или verified assessment, соответствующий блок показывает inline error с понятным сообщением и retry.

   3. Если Monaco не загрузился в assessment, code answer должен fallback to ordinary textarea для ввода Python-кода.

   4. Если Pyodide не загрузился, run/debug actions disabled с inline error и retry.

4. Offline policy:

   1. Полноценная offline-работа не является обязательной в v1.

   2. Browser cache может помочь повторным открытиям, но это не считается гарантией.

   3. Self-contained export не входит в v1.

5. Media policy:

   1. Embedded images/videos задаются URL-only.

   2. Builder не копирует media рядом с HTML и не встраивает base64 в v1.

   3. Недоступность external media не должна ломать весь документ.

6. Theme runtime:

   1. Theme toggle доступен в верхней панели generated HTML.

   2. Выбор темы сохраняется локально в browser storage.

   3. Theme не сохраняется в student answer state.

7. Python runtime:

   1. Pyodide работает в Web Worker.

   2. Каждый run/test выполняется в fresh namespace.

   3. Runtime задаёт timeout и output cap для stdout/stderr.

   4. Timeout, excessive output, runtime error и syntax error показываются в output panel понятным сообщением.

   5. Доступные Python-пакеты: stdlib + allowlist, заданный в project metadata.

   6. Arbitrary `micropip` без allowlist не поддерживается в v1.

8. Student assessment save runtime:

   1. Assessment HTML показывает действие `Разрешить автосохранение`.

   2. При нажатии студент выбирает текущий HTML-файл через File System Access API.

   3. После успешного permission accepted answers записываются в этот HTML-файл on submit.

   4. Если File System Access API недоступен, permission не дан или запись не удалась, показывается persistent status banner.

   5. Banner объясняет, что перенос между устройствами не гарантирован, пока HTML не обновлён.

   6. Fallback action: скачать updated HTML с текущим `pb-answer-state`.

   7. Verified-builder принимает fallback так же, как обычный student HTML: входы остаются student HTML + teacher key JSON.

9. Student assessment multi-tab:

   1. Runtime не блокирует несколько вкладок.

   2. Last write wins.

   3. UI не обязан строить lock-систему в v1.



Shared content blocks and rich-text model

1. Общие блоки:

   1. Paragraph.

   2. Numbered list.

   3. Bulleted list.

   4. Table.

   5. Embedded image by URL.

   6. Embedded video by URL.

   7. Special block.

   8. Monaco Editor block.

   9. Question block только для theory и assessment.

2. Paragraph:

   1. Paragraph body хранится как Inline AST.

   2. Paragraph поддерживает plain text, bold, italic, underline, inline code, link.

3. Lists:

   1. List item text хранится как Inline AST.

   2. List items поддерживают plain text, bold, italic, underline, inline code, link.

   3. Вложенные списки не являются обязательными для v1.

4. Tables:

   1. Table v1 поддерживает простые ячейки inline-text.

   2. Table поддерживает строки, колонки, optional header row.

   3. Cell content поддерживает Inline AST.

   4. Block content внутри table cell не поддерживается в v1.

5. Media:

   1. При добавлении image/video builder показывает поле URL.

   2. Esc отменяет добавление.

   3. Enter подтверждает добавление.

   4. URL проходит sanitization.

6. Special block:

   1. Special block строится по единому шаблону: `block-label`, h3 title, body.

   2. Единый набор labels доступен во всех document types:

      `Определение термина`, `Важный момент`, `Подсказка`, `Верный подход`, `Неверный подход`.

   3. Выбор label меняет style.

   4. h3 title поддерживает plain text и inline code.

   5. Body может содержать paragraph, numbered list, bulleted list, table, image, video, Monaco block.

   6. Если body пустое, внутри показывается plus-кнопка для добавления первого body block.

7. Monaco Editor block:

   1. Monaco block имеет title, role, mode, default code, action buttons, editor area и optional output panel.

   2. Roles: example, self-run, graded.

   3. Example/self-run blocks не влияют на section status.

   4. В одной секции может быть только один graded Monaco block.

   5. Допустимые modes для обычного example/self-run block: readonly, readonly+run, edit+run.

   6. Editable без run не является обязательным режимом v1.

   7. Reset возвращает код к author default.

   8. Copy копирует текущий код.

   9. Run выполняет код через Pyodide, если включён.

   10. Output panel показывается только если run/debug включён или есть runtime output/error.

   11. В builder при hover над editor area показываются toggles run/readonly/debug where applicable.

   12. Inactive toggle выглядит прозрачным с пунктирной границей. Active toggle выглядит непрозрачным с solid border.

   13. В generated HTML buttons в правом верхнем углу: reset if editable, run if enabled, copy, debug if enabled for practice.

8. Question block:

   1. Question block используется в theory и assessment.

   2. Question block строится по шаблону special block: `block-label`, h3 title, body.

   3. Question labels: `Вопрос с одним ответом`, `Вопрос с несколькими ответами`, `Открытый вопрос`.

   4. h3 title поддерживает plain text и inline code.

   5. Body может содержать paragraph, numbered list, bulleted list, table, image, video, Monaco example/self-run block.

   6. Question body не содержит graded Monaco block.

   7. Single/multiple answer question содержит answer options.

   8. Option text поддерживает plain text, bold, underline, inline code.

   9. Text input question содержит answer input и `Ответить`.

   10. Theory question показывает immediate feedback и не сохраняет state между сессиями.

   11. Assessment question сохраняет accepted answer только после explicit submit.

9. Autotest model:

   1. Autotests хранятся структурно.

   2. Test types: input/print, function, class/scenario.

   3. Input/print test хранит input text, expected output, hidden flag, timeout override optional.

   4. Function test хранит function call expression in constrained format и expected value.

   5. Class/scenario test хранит controlled Python scenario body.

   6. Builder генерирует executable harness из structured schema.

   7. В practice tests могут храниться в HTML. Hidden скрывает details in UI only.

   8. В assessment tests хранятся только в teacher key JSON.

10. Text answer matching:

   1. Text input answers сравниваются через normalization + allowed variants.

   2. Normalization включает trim и нормализацию переводов строк/пробелов.

   3. Case-insensitive matching может быть опцией конкретного question.

   4. Regex/rule-based matching не является обязательным для v1.

   1. Correct result подсвечивается зелёным.

   2. Incorrect result подсвечивается красным.

   3. Missing/mismatch result получает нейтральный статус `Ожидает проверки`.

   4. Для text input под student answer показывается список accepted correct variants, если verified output раскрывает keys.

   1. Показывает student code.

   2. Показывает autotest results.

   3. Показывает expected code, если verified output раскрывает keys.

   4. Ниже может быть teacher comment.

   1. Section frame green for `Верно`.

   2. Section frame red for `Неверно`.

   3. Neutral/warning frame for `Ожидает проверки`.

   4. Sidebar item показывает corresponding status.

   1. Verified HTML предназначен для обратной связи студенту после проверки.

   2. Verified HTML может содержать correct answers, expected code и test results.

   3. Student assessment HTML до проверки не содержит эти данные.

   1. Inputs: student assessment HTML + teacher key JSON + output verified HTML path.

   2. Builder validates PB manifest, document type, `documentId`, `contentHash`.

   3. Builder reads `pb-answer-state` from student HTML.

   4. Builder runs answer matching for questions.

   5. Builder runs code autotests through Pyodide/harness.

   6. If answer not found, matching key not found or test cannot be run, section status is `Ожидает проверки`.

   2. Teacher can manually mark section `Верно` or `Неверно`.

   3. Manual override wins over auto-result.

Save/open/autosave/recovery rules

1. Manual save:

   1. `Save` writes current project to configured path.

   2. `Save as...` asks for a new path.

   3. Theory/practice save writes one HTML.

   4. Assessment save writes student HTML and teacher key JSON.

   5. Verified save writes verified HTML.

   6. Assessment save is atomic across HTML + key JSON as far as possible: write temp files, replace only after both are ready, report failure clearly.

2. Builder autosave:

   1. Builder autosaves every 5 minutes.

   2. Builder keeps 10 autosave slots.

   3. Autosave slots are stored рядом с проектом.

   4. If all slots are used, oldest slot is overwritten.

   5. Autosave does not replace the main HTML unless user explicitly saves.

   6. On autosave builder shows top toast/banner with autosave status.

3. Recovery:

   1. On app start/open builder checks for recovery snapshots.

   2. If snapshots exist, builder prompts with snapshot list, timestamp, document type and source path.

   3. User can restore snapshot, ignore it or delete obsolete snapshot.

   4. Builder must not silently overwrite main project with autosave snapshot.

4. Open:

   1. Builder opens only HTML with valid PB manifest.

   2. Builder rejects missing/invalid `appId`, `documentType`, `documentId`, `schemaVersion`, ids or metadata blocks.

   3. Builder rejects unsupported schemaVersion in v1.

   4. Builder reports actionable error and does not mutate the source file during failed open.

5. Recent projects:

   1. Successful open/save updates recent projects.

   2. Missing files in recent list are shown as unavailable or pruned after confirmation.

6. Link/media checking:

   1. Save does not require network.

   2. Builder provides optional `Check links/media` command.

   3. Check command reports unreachable media/link URLs but does not automatically delete or rewrite them.

7. Validation:

   1. Errors block save/export.

   2. Warnings allow save/export.

   3. Blocking errors include missing manifest fields, duplicate ids, broken section refs, broken anchors, missing graded config, assessment graded item without key data, invalid URL scheme, invalid artifact pairing.

   4. Empty optional content, empty feedback, unfinished media URL and similar authoring gaps are warnings unless they break runtime.



Validation, testing, and quality gate

1. Quality gate for feature work:

   1. Unit tests for document model, commands, validation, serialization, deserialization and artifact naming.

   2. Unit tests for grading rules: exact set, text normalization, code all-tests-pass semantics and missing-answer handling.

   3. Snapshot tests for generated HTML structure, metadata blocks and absence of assessment keys in student HTML.

   4. E2E tests for generated HTML runtime using browser automation.

   5. E2E tests for key builder flows: create, save, open, add blocks, undo/redo, assessment submit fallback, verified creation.

2. Required E2E scenarios:

   1. Create theory, add section, add paragraph, save, reopen exact authoring state.

   2. Create practice, add graded code task, run visible and hidden tests, verify status UI.

   3. Create assessment, add choice/text/code graded task, save HTML + key JSON, confirm student HTML does not contain key data.

   4. Open assessment as student HTML, submit answer, write `pb-answer-state` to HTML or download updated HTML fallback.

   5. Create verified assessment from student HTML + key JSON, confirm score, statuses, comments and manual override.

   6. Runtime failure scenario: Monaco failure in assessment uses textarea fallback.

   7. Runtime failure scenario: theory/practice/verified show inline error with retry.

3. Performance targets:

   1. Builder interactions should not have noticeable lag for normal lesson sizes.

   2. Generated HTML should show pre-rendered content quickly before heavy feature modules finish loading.

   3. Monaco and Pyodide must load lazily.

   4. Long Python execution must be stopped by timeout.

4. Security tests:

   1. URL sanitization blocks dangerous schemes.

   2. Raw HTML/JS insertion is not accepted by content editors.

   3. Assessment student HTML does not include correct answer values, test bodies or expected code.

   4. Verified HTML may include keys only after teacher-side verification output is created.

5. Logging:

   1. Builder writes local logs in root builder logs directory.

   2. Logs include errors for save/open/runtime bridge failures.

   3. Logs do not include student answer content by default.

   4. No data is sent over network automatically.



Out of scope for v1

1. Server-side learning platform, accounts, classrooms, cloud sync or remote submission.

2. Automatic telemetry, crash reporting or analytics.

3. Raw HTML/JS blocks in lesson content.

4. Full mobile-first authoring experience.

5. Guaranteed full offline generated HTML.

6. Self-contained export with all CSS/JS/Monaco/Pyodide embedded.

7. Automatic builder auto-update.

8. Full localization system for builder and generated HTML.

9. Arbitrary Python package installation through unrestricted `micropip`.

10. Regex/rule-based text answer grading.

11. Partial credit scoring.

12. Multiple graded tasks in one section.

13. Practice question blocks.

14. Full WCAG AA audit.

15. Custom CSS/theme authoring.

16. Block content inside table cells.

17. Automatic schema migrations from older document versions.



Risk register and tradeoffs

1. Risk: потеря ответов студента при записи assessment HTML.

   Decision: assessment writes accepted answers on submit through File System Access API when available.

   Fallback: persistent warning banner + download updated HTML.

   Residual tradeoff: browsers without reliable file write support require explicit download action.

2. Risk: student HTML accidentally раскрывает правильные ответы или автотесты.

   Decision: assessment key data хранится только в teacher key JSON.

   Fallback: validation/snapshot tests проверяют отсутствие key data в student HTML.

   Residual tradeoff: practice tests остаются открытыми в HTML, а `Hidden` является только UI-флагом.

3. Risk: HTML становится слишком тяжёлым из-за pre-rendered DOM + full metadata duplication.

   Decision: v1 prioritizes exact reopen over minimal metadata size.

   Fallback: keep media URL-only, lazy-load heavy libs, keep metadata readable but compact.

   Residual tradeoff: file can be larger than JSON-only render, but target remains up to 1-2 MB for normal content.

4. Risk: GitHub Pages runtime update ломает старые generated HTML.

   Decision: generated HTML pins runtime version and uses SRI where possible.

   Fallback: manual releases and stable runtime paths.

   Residual tradeoff: old HTML will not automatically receive runtime fixes unless regenerated or runtime version policy explicitly allows patching.

5. Risk: `file://` режим ограничивает browser APIs.

   Decision: target Edge/Chrome and test generated HTML in double-click file mode.

   Fallback: runtime shows inline errors and uses assessment textarea/download fallback where needed.

   Residual tradeoff: unsupported browsers are not primary target.

6. Risk: WYSIWYG UI becomes hard to discover because many operations are hover/right-click.

   Decision: combine context menu and hover insertion with explicit selected-block actions and inspector.

   Fallback: keyboard shortcuts and visible focus states.

   Residual tradeoff: visual surface is slightly less minimal than pure hover-only design.

7. Risk: custom rich-text editor is complex.

   Decision: limit rich-text model to Inline AST and small mark set.

   Fallback: use Selection API wrappers only for browser-specific behavior.

   Residual tradeoff: advanced rich-text features are out of scope.

8. Risk: Pyodide startup and execution are slow.

   Decision: lazy-load Pyodide on first run/debug and run in Worker with fresh namespace.

   Fallback: show loading state and timeout/output cap.

   Residual tradeoff: first run can still be noticeably slower than later runs.

9. Risk: code execution leaks state between runs/tests.

   Decision: Worker + fresh namespace per run/test.

   Fallback: recreate worker only if namespace isolation proves insufficient.

   Residual tradeoff: fresh worker per run would be cleaner but slower, so it is not default.

10. Risk: verified assessment uses wrong key JSON.

    Decision: pairing uses `documentId + contentHash`.

    Fallback: mismatch blocks or requires explicit teacher confirmation.

    Residual tradeoff: contentHash policy must be carefully tested when author edits assessment after distribution.

11. Risk: one section with multiple questions/code tasks makes scoring ambiguous.

    Decision: v1 allows exactly one graded task per section.

    Fallback: author can split content into multiple sections.

    Residual tradeoff: section structure may be more granular than lesson narrative.

12. Risk: autosave overwrites intentionally saved project.

    Decision: builder autosave slots live рядом с проектом and do not replace main HTML.

    Fallback: recovery prompt on start/open.

    Residual tradeoff: project folder contains autosave files.

13. Risk: no schema migrations makes older files hard to open.

    Decision: v1 supports only current schema.

    Fallback: strict open error message explains unsupported version.

    Residual tradeoff: migration work is deferred until schema stabilizes.

14. Risk: external media links disappear or become unavailable.

    Decision: media is URL-only to keep HTML light.

    Fallback: optional Check links/media command.

    Residual tradeoff: generated HTML depends on external media availability.

15. Risk: no telemetry reduces support visibility.

    Decision: no telemetry by default.

    Fallback: local logs in root builder logs directory can be inspected manually.

    Residual tradeoff: diagnosing user-specific failures requires manual log sharing.

16. Risk: WYSIWYG feel conflicts with first vertical slice scope.

    Decision: WYSIWYG feel is non-negotiable for core slice.

    Fallback: first slice limits block variety instead of lowering interaction quality.

    Residual tradeoff: fewer block types may be complete in the first milestone.

17. Risk: assessment textarea fallback gives worse code editing experience.

    Decision: fallback is only for Monaco load failure in assessment, to preserve ability to submit.

    Fallback: inline retry remains available.

    Residual tradeoff: fallback lacks Monaco features but prevents data-entry blockage.

18. Risk: manual override in verified assessment hides auto-check bugs.

    Decision: manual wins, but original auto-result remains in review metadata.

    Fallback: UI can display auto-result history/details to teacher.

    Residual tradeoff: final score may differ from automatic result by teacher intent.

19. Risk: URL sanitization blocks legitimate local teaching resources.

    Decision: safe allowlist protects students from dangerous schemes.

    Fallback: author can use hosted `https` resources or future explicit trusted-local policy.

    Residual tradeoff: some local/private URLs may require adjustment.

20. Risk: first milestone full vertical slice is broad.

    Decision: vertical slice includes all four output flows but only core blocks + one graded flow.

    Fallback: postpone non-core blocks like complex tables, media polish and advanced feedback editing.

    Residual tradeoff: early product demonstrates architecture and persistence before full content-block completeness.
